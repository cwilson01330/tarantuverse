## P0 ÔÇö Account deletion is broken for real users (FIX FIRST)

**Endpoint:** `apps/api/app/routers/auth.py::delete_account` (`DELETE /auth/me`)

```python
db.delete(current_user)   # "CASCADE will handle related data"  ÔćÉ it won't, via ORM
db.commit()
```

**Why it breaks:** `User.animals`, `User.tarantulas`, `User.inverts`,
`User.scorpions`, `User.scorpion_colonies`, `User.broods` are all defined as
**child-side backrefs without `passive_deletes=True`**. Their `user_id` columns
are `ondelete="CASCADE"` at the DB level **and** `nullable=False`. On
`db.delete(user)`, SQLAlchemy ignores the DB cascade (because `passive_deletes`
isn't set), loads each collection, and tries to `UPDATE ... SET user_id = NULL`
to disassociate them Ôćĺ NOT NULL violation Ôćĺ `IntegrityError` Ôćĺ 500.

This is the **same root cause** as the invert-delete 500 fixed 2026-06-11
(see [[feedback-polymorphic-backref-passive-deletes]]). It works for empty
accounts (no children to null) which is why it likely passed a quick test, but
fails the moment a user owns one animal.

**Confirmed impact (prod, 2026-06-12):** 37 users own inverts, 1 owns animals.
Any of those 38 deleting their account gets a 500.

**Fix:** add `passive_deletes=True` to the User-side of these relationships so
the DB `ON DELETE CASCADE` does the work. Because they're declared as backrefs
on the child models, either convert to `backref(backref("...", passive_deletes=True))`
or (cleaner) declare them explicitly on `User` with `passive_deletes=True` and
drop the child backref name. Must cover: tarantulas, inverts, scorpions,
scorpion_colonies, broods, animals. Same FK/cascade facts already verified.

**Also latent (lower urgency):** `shed_logs.animal_id` and `weight_logs.animal_id`
are NOT NULL with backrefs lacking `passive_deletes`. The per-animal delete route
(`routers/animals.py`) masks this by manually bulk-deleting children before
`db.delete(animal)`, so direct animal deletion works ÔÇö but the pattern is fragile
and inconsistent with how inverts/tarantulas delete. Fold these into the same fix.

**Verification after fix:** create a throwaway account, add a reptile with a
weight log, delete the account Ôćĺ expect 200, and confirm the rows are gone.

## Still open ÔÇö triaged

### Launch-relevant
- **Account deletion backend (P0 above).** Re-opens the store blocker in practice.
- **Data export on HV (both surfaces).** Absent. TV has it; reptile keepers' export
  must contain reptile data. Compliance-relevant (data portability) and a real
  parity gap. The TV exporter is taxon-specific, so this needs a reptile-aware path.
- **Legal pages still hosted on tarantuverse.com** (`/herpetoverse/privacy-policy`,
  `/herpetoverse/terms`). Works, but they arguably belong on herpetoverse.com before
  a public launch; the App Store listing also needs a privacy URL.

### Deferred by design (not blockers)
- **Theme/appearance UI** ÔÇö `ThemeContext` is dark-only by intent; presets deferred to Phase 3.6.
- **Onboarding flow (mobile)** ÔÇö no welcome screens; parity polish.
- **Achievements on mobile** ÔÇö web has it, mobile has none; also categories still
  read tarantula-flavored (no shed/weight/clutch milestones yet).
- **Native registration on HV web** ÔÇö links out to tarantuverse.com/register by
  design (shared account). HV mobile registers natively ÔÇö the surfaces are
  inconsistent; pick one stance and apply it both places.
- **Community, analytics, shed-prediction, global search, referral, enclosures UI,
  admin** ÔÇö all explicitly v1.1+ per prior decisions.

## Recommended sequencing

1. **Fix `DELETE /auth/me`** (P0 ÔÇö passive_deletes on User collections + shed/weight). Small, high-impact, unblocks the store for real.
